# Internationalization (i18n)

The Tikka client ships with English (`en`) and Spanish (`es`) translations that
are loaded at runtime by [`i18next`](https://www.i18next.com/) via
[`react-i18next`](https://react.i18next.com/). Source of truth lives in
[`client/public/locales/`](../public/locales).

This document explains how to add new translatable strings and how locale
parity is enforced.

## Directory layout

```
client/public/locales/
├── en/
│   └── common.json         # English strings (the reference locale)
└── es/
    └── common.json         # Spanish translations
```

Namespace files (`common.json`) hold a nested JSON tree. Keys are referenced
in React code via the `t()` function and the `useTranslation` hook, for
example:

```tsx
const { t } = useTranslation();
return <h1>{t('home.catchNextOpportunity')}</h1>;
```

The backend load path is configured in [`client/src/i18n.ts`](../src/i18n.ts):

```ts
backend: {
  loadPath: '/locales/{{lng}}/{{ns}}.json',
},
supportedLngs: ['en', 'es'],
fallbackLng: 'en',
```

`en` is the **reference locale**: every other locale must contain the same
keys (and the same value *shape*) as `en/common.json`.

## Adding a new translation key

1. **Add the key to English first** (`client/public/locales/en/common.json`).
   Keep the surrounding namespace logically grouped:

   ```jsonc
   {
     "raffle": {
       "status": "Live"          // ← new key
     }
   }
   ```

2. **Mirror the key in every other locale** (`es/common.json`, plus any future
   locales such as `fr/`, `de/`, …). Translate the value — do not copy English
   text verbatim — but use the identical dotted path and matching value type
   (string ↔ string, object ↔ object).

3. **Reference the key in code** using the fully-qualified path:

   ```tsx
   <span>{t('raffle.status')}</span>
   ```

   Avoid hard-coding user-facing English copy in components; route it through
   `t()` so the translation system can swap it at runtime.

4. **Run the locale check locally**:

   ```bash
   pnpm run check:locales
   ```

   The check exits non-zero if any locale is missing keys, has extra keys
   not present in `en`, or has a value whose type differs (e.g. object ↔
   string).

5. **Open a PR**. The same check runs automatically in CI (see below).

## How parity is enforced

Two layers run the same comparison logic so drift cannot sneak in:

| Layer | File | When it runs |
|---|---|---|
| Standalone CLI | [`scripts/check-locales.js`](../scripts/check-locales.js) | `pnpm run check:locales`, `pnpm test*`, `pnpm build` (via `pretest` / `prebuild` hooks) |
| Vitest spec | [`src/i18n/locales.spec.ts`](../src/i18n/locales.spec.ts) | `pnpm test`, CI `Run unit tests` job |

Both:

- treat **`en` as the reference** (mirroring `i18n.ts` → `fallbackLng`),
- enumerate every `*.json` namespace under each locale directory,
- flatten nested objects into `dotted.path` keys,
- report four classes of drift:
  - **MISSING** — key exists in `en` but not in this locale,
  - **EXTRA** — key exists in this locale but not in `en` (often a leftover),
  - **MISMATCH** — the same key is a string in `en` and an object (vice-versa)
    in this locale,
  - empty string in a non-reference locale **and** a non-empty value in `en`
    is reported as **MISSING**.

The standalone script also supports `NO_COLOR=1` for clean CI logs.

## CI behaviour

`.github/workflows/ci.yml` runs the following for the `client` package:

```
install → lint → Run unit tests → Build
```

`Run unit tests` invokes `pnpm run test:unit`, which expands to
`pretest` → `check:locales` → `vitest --run`. This means a missing locale
key fails both the standalone check and the vitest assertion before the
build step ever starts.

## Reviewing findings

When the check fails, the output looks like:

```
❌ es — 2 finding(s) (46 keys)
   namespace: raffle
   MISSING  raffle.status
   MISMATCH raffle.status (ref: object, other: leaf)
   …
```

To resolve:

1. Open the offending JSON file referenced in the finding.
2. Add the missing key in the same namespace, translating the value.
3. Re-run `pnpm run check:locales` until the output is all green:

```
✅ en — 100% complete (54 keys across 1 namespace(s))
✅ es — 100% complete (54 keys)
```

## Adding a new language

1. Create `client/public/locales/<code>/common.json` with the full key tree
   from `en/common.json` translated.
2. Add the new code to `supportedLngs` in `client/src/i18n.ts`:

   ```ts
   supportedLngs: ['en', 'es', '<code>'],
   ```

3. Update any language switcher UI if one exists.
4. Run `pnpm run check:locales` — it will list the new locale automatically
   once the directory exists.

The locale completeness check then guarantees that future additions to
`en/common.json` propagate to the new locale before they can reach production.

## Reference

- `client/src/i18n.ts` — runtime i18n configuration
- `client/scripts/check-locales.js` — standalone parity checker
- `client/src/i18n/locales.spec.ts` — vitest parity assertion
- `client/public/locales/<lng>/<ns>.json` — translation source of truth
